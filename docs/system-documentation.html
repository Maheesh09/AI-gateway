System Architecture
What this system does
Every HTTP request from an external client passes through the gateway. The gateway checks who you are (auth), whether you've sent too many requests (rate limit), then forwards you to the right upstream service (proxy). After proxying, it asynchronously submits the request to an AI analysis queue. A separate worker process dequeues jobs, detects anomalies, calls the Claude API, and stores alerts in PostgreSQL.

Step-by-Step Request Flow
Client sends HTTP request to GET /api/your-service/endpoint

The client must include either Authorization: Bearer <JWT> or X-API-Key: gw_<hex>.

Auth Middleware (middleware/auth.go) intercepts

If Bearer token → decode with HMAC-SHA256, extract key_id + owner_id from claims.
If X-API-Key → SHA-256 hash the raw key, look up api_keys.key_hash in PostgreSQL, check is_active and expires_at.
On success, inject both IDs into the request context.

Rate Limiter Middleware (middleware/ratelimit.go) checks Redux quota

Executes a Lua script atomically on Redis sorted set keyed rate_limit:<key_id>. Removes entries older than the window, counts remaining, adds current request if under limit or returns -1 (blocked). Sets X-RateLimit-* headers on the response.

Proxy Router (proxy/router.go) matches the request path

Loads active routes from PostgreSQL via RouteRepo.ListActive(). Matches the URL path against each route's path_pattern using wildcard matching. Also checks the HTTP method against the route's allowed_methods list. Returns the matching ProxyRoute.

Upstream Proxy (proxy/upstream.go) forwards the request

Creates an httputil.ReverseProxy with a custom Director that rewrites the URL to the route's target_url, optionally strips the path prefix, applies a per-route timeout, and removes internal headers (X-API-Key, Authorization) before forwarding.

Gateway wraps the ResponseWriter to capture status code

A thin statusWriter struct wraps http.ResponseWriter and intercepts WriteHeader() to record what status code the upstream returned.

Gateway enqueues an async analysis job

After the upstream responds, calls ai.EnqueueAnalysis() with an AnalyzePayload containing API key ID, route ID, method, path, status code, latency, and IP. This serialises to JSON and pushes to the asynq default queue in Redis. This is fire-and-forget; if Redis is unavailable the error is swallowed so the client is never penalised.

Worker dequeues job (ai/worker.go): HandleAnalyzeTask

Step a: deserialise the JSON payload.
Step b: call LogRepo.Insert() to persist the request to request_logs.
Step c: call Detector.Evaluate(); if no rule fires, exit early.
Step d: if a rule fires, call Analyzer.Analyze() (Claude API).
Step e: call AlertRepo.Insert(); if auto_blocked=true also disables the API key in the same DB transaction.

internal/db — Database Connections
db/postgres.go

Creates and verifies a pgxpool.Pool — a connection pool to PostgreSQL using the high-performance pgx v5 driver.

FunctionWhat it does
NewPool(url)Parses config URL, sets pool limits (min=2, max=10, idle=30min), pings the DB to verify connectivity, returns the pool or an error.

The pool is shared across all repository types. pgxpool is safe for concurrent use across goroutines.

db/redis.go

Creates a Redis client and defines the sliding-window Lua script used by the rate limiter.

FunctionWhat it does
NewRedisClient(url)Parses redis:// URL via redis.ParseURL, creates client, pings Redis. Returns the client or an error.
SlidingWindowScriptA pre-loaded Redis Lua script: removes outdated sorted-set members, counts current, adds if under limit, returns count or -1.
Why Lua?
Redis executes Lua scripts atomically — no race condition between checking the count and adding the new request. This is the correct way to implement sliding-window rate limiting.

internal/proxy — Reverse Proxy Engine
proxy/router.go

Loads active routes from the database and matches incoming requests to them using wildcard path patterns.

FunctionWhat it does
NewRouter(routeRepo)Constructor. Holds a reference to the route repository.
Match(ctx, r)Calls routeRepo.ListActive() to get all active routes. For each route, uses wildcard path matching and allowed-method checking. Returns the first matching route.
Hot-Reload
Routes are fetched on every request, which means you can add or modify routes via the admin API and they take effect immediately without restarting the gateway.
proxy/upstream.go

Creates a configured httputil.ReverseProxy for a matched route that handles forwarding and header sanitisation.

FunctionWhat it does
NewUpstreamProxy(route)Parses the target URL, creates a ReverseProxy with custom Director and Transport. Returns the proxy or an error if the URL is invalid.
Director funcRewrites the request URL to point at the upstream target. If StripPrefix=true, removes the matched path prefix. Deletes sensitive headers before forwarding.
TransportCustom http.RoundTripper with a TimeoutMs-controlled deadline — prevents slow upstreams from blocking gateway goroutines.

Fault tolerance
If the Claude API call fails (network error, quota exceeded, etc.), the worker falls back to creating a MEDIUM rule-only alert instead of returning an error. The log entry is always persisted regardless.

cmd/ — Binary Entrypoints
cmd/gateway/main.go

The HTTP gateway binary. Wires everything together and starts the HTTP server.

Startup sequence:

  1. Load config from .env
  2. Open PostgreSQL pool
  3. Open Redis client
  4. Create all repositories
  5. Create asynq client
  6. Create auth + rate limit middleware
  7. Create admin handlers
  8. Build chi router with all routes
  9. Start http.ListenAndServe

Also defines statusWriter (response wrapper) and extractIP() (strips port from RemoteAddr).

cmd/worker/main.go

The background AI analysis worker binary. Runs as a separate process consuming jobs from Redis.

Startup sequence:

  1. Load config from .env
  2. Open PostgreSQL pool
  3. Create LogRepo + AlertRepo
  4. Create Detector + Analyzer
  5. Create Worker (assembles the pipeline)
  6. Start asynq server (concurrency=5)
  7. Register HandleAnalyzeTask handler
  8. Block on srv.Run(mux)

Shuts down gracefully on SIGTERM / SIGINT — asynq handles this internally.

Improvements & Fine-Tuning
🎯 Per-Key Rate Limits

Currently all keys share the default RPM. To support per-key limits:

  • In ratelimit.go, look up key.RateLimitRPM from the context (requires storing it there in auth middleware)
  • Pass that as the limit to check() instead of rl.defaultRPM
  • Update auth middleware to also set ContextKeyRPM from the DB record
⚡ Route Caching

RouteRepo.ListActive() hits PostgreSQL on every request. At high traffic:

  • Cache routes in Redis with a short TTL (5–10s)
  • Or cache in-process with an atomic.Value + background refresh goroutine
  • Invalidate on every admin PUT/DELETE to routes
🤖 AI Detector Tuning
  • Make thresholds configurable via env vars or a config table in PostgreSQL
  • Add more rules: unusual status code distribution, path scanning patterns, time-of-day anomalies
  • Tune detection window from 5min to 1min for tighter real-time detection
  • Add a confidence score to the Claude response and only block at >80% confidence
📊 Observability
  • Expose Prometheus metrics using prometheus/client_golang (already in go.mod)
  • Track: request count per route, p99 latency, rate limit hit rate, alert count by severity
  • Add structured JSON logging with go.uber.org/zap (already in go.mod)
